# AI Lifelog / note Export Design

更新日: 2026-03-15

## 1. 目的

この設計は、`フィルム体験のあとにライフログの下書きをAIで作り`、`noteに持っていきやすい形で書き出す` ためのものです。

狙いは次の3つです。

- 写真を撮ったあとに、文章を書き始める負荷を下げる
- フィルムロール単位の体験を「1本の記録」として残しやすくする
- 将来の「動物とるカメラ」にも同じAI基盤を流用できるようにする

## 2. 前提方針

- AI連携は `自前バックエンド経由` にする
- `ChatGPT / Claude のユーザーログインをアプリに持ち込まない`
- 初期段階では `noteへ直接投稿` はやらない
- 初期段階では `note向けに整った本文を書き出す` ところまでをやる
- 画像自体は最初は送らず、まずは `テキスト + メタ情報` だけで生成する

## 3. MVPの体験

### 3.1 生成タイミング

- フィルム現像完了後
- または `JournalScreen` の保存前

まずは `現像完了後` が自然です。ロール全体が揃っていて、1本の記録として文章化しやすいためです。

### 3.2 ユーザーフロー

1. ユーザーがフィルムを現像する
2. アプリが `AIでライフログ下書きを作る` を提案する
3. ユーザーが生成を実行する
4. バックエンドがAIへ要約・本文生成を依頼する
5. アプリに `タイトル / 導入 / 本文 / タグ候補` を返す
6. ユーザーがアプリ内で編集する
7. `note向けに書き出す`

### 3.3 note向け書き出し

初期は次の3つを提供します。

- `Markdownとしてコピー`
- `プレーンテキストとしてコピー`
- `共有シートでnoteへ渡しやすい本文を書き出す`

ここでいう `noteに書き出せる感じ` は、まずは `noteエディタに貼るだけで使える本文を作る` ことを意味します。

## 4. 出力イメージ

AIは自由作文ではなく、次の構造で返します。

- タイトル
- サブタイトル
- 導入 2〜4文
- 本文セクション 2〜4個
- しめの一文
- ハッシュタグ候補 3〜6個
- 短いSNS用要約

本文のトーンは次の2種類から開始します。

- `日記寄り`
- `note記事寄り`

### 4.1 note記事寄りの例

- タイトル: `上野で過ごした午後、レッサーパンダとやわらかい光`
- 導入: その日の空気感を短くまとめる
- 本文1: 場所とテーマ
- 本文2: 印象に残った被写体
- 本文3: 撮っていて感じたこと
- しめ: 次に見返したい理由

## 6. アプリ内データモデル

既存の `FilmSession` と `Photo` はそのまま活かし、AI生成結果を新しいエンティティで持ちます。

### 6.1 追加モデル

- `provider`

- `tone`
- `title`
- `subtitle`
- `intro`
- `body_markdown`
- `body_plain_text`
- `hashtags_json`
- `social_summary`
- `source_snapshot_json`
- `created_at`

### 6.2 source_snapshot に含めるもの

- セッションタイトル
- 場所

これを保存しておくと、あとで `同じ材料から再生成` ができます。

## 7. バックエンド設計

### 7.1 役割

バックエンドは次を担当します。

- アプリ認証
- 利用制限
- AIプロンプト生成
- OpenAI / Anthropic 呼び分け
- 生成ログ
- 将来の課金や使用量管理

### 7.2 エンドポイント

#### `POST /v1/ai/lifelog-drafts`

用途:

- ライフログ下書きの新規生成

リクエスト:

- `session_id`
- `tone`
- `export_target`
- `session_context`
- `photo_contexts`

レスポンス:

- `draft_id`
- `title`
- `subtitle`
- `intro`
- `body_markdown`
- `body_plain_text`
- `hashtags`
- `social_summary`
- `provider`
- `model`

#### `POST /v1/ai/lifelog-drafts/{draft_id}/regenerate`

用途:

- 同じ素材で再生成

#### `POST /v1/ai/lifelog-drafts/{draft_id}/refine`

用途:

- `もっと短く`
- `note向けにする`
- `やわらかくする`
- `観察寄りにする`

#### `GET /v1/ai/lifelog-drafts/{draft_id}`

用途:

- 保存済み下書きの取得

## 8. プロンプト設計

### 8.1 基本方針

- 画像の説明を断定しない
- ユーザーが書いたメモを主材料にする
- セッション全体を `一本の体験` としてまとめる
- 大げさすぎる感情表現は避ける
- `noteに貼って自然` な文体にする

### 8.2 システムプロンプトの方向

- 写真日記アシスタントとして振る舞う
- 断定ではなく観察ベースで書く
- 被写体名は入力にあるものだけを使う
- 事実と印象を混ぜるが、作り込みすぎない
- 体験が主役で、AIらしさは抑える

### 8.3 ユーザー入力整形

アプリからはそのままDB内容を投げず、バックエンドで整形してからプロンプトへ入れます。

整形例:

- `session_overview`
- `location_summary`
- `theme_summary`
- `photo_notes`
- `timeline_summary`

## 9. note書き出し仕様

## 9.1 初期スコープ

初期は `note投稿API連携` を前提にしません。

出力形式は:

- `Markdown`
- `Plain Text`
- `Share Text`

### 9.2 Markdownテンプレート

```md
# {title}

{subtitle}

{intro}

## その日のこと

{body_section_1}

## 印象に残った場面

{body_section_2}

## あとから見返したいこと

{body_section_3}

{closing}

{hashtags}
```

### 9.3 将来拡張

将来は次を検討できます。

- note向け専用トーン
- 見出し数の自動調整
- 写真キャプション候補
- セッションのインデックスシート画像を含む書き出し

## 10. セキュリティ / プライバシー

- AIへ送る対象は最初は `テキスト情報のみ`
- 位置情報は `施設名レベル` を基本とする
- 緯度経度は送らない
- ユーザーが手入力したメモを送る前に明示
- `AI生成内容には誤りが含まれることがあります` を表示
- 保存前にユーザー編集を必須にする

## 11. 将来の「動物とるカメラ」への流用

今回の設計は `写真ロールを文章化する基盤` として切り出すため、そのまま動物カメラにも使えます。

流用できる要素:

- AIプロバイダ呼び分け
- 下書き生成API
- Markdown出力
- トーン制御
- source snapshot

動物カメラ側で追加するもの:

- 観察した動物種
- 行動タグ
- 観察メモ
- 展示エリア

つまり `FilmSession専用機能` ではなく、将来的には `CaptureSession` 系へ一般化できる設計にしておくのがよいです。

## 12. MVP実装順

### Phase 1

- バックエンドに `lifelog-drafts` API を作る
- アプリで `AI下書き生成` ボタンを置く
- 文章を返して編集できるようにする
- `コピー / 共有` を付ける

### Phase 2

- note向けテンプレート選択
- 再生成 / リライト
- タグ候補

### Phase 3

- 画像を含む高度な生成
- 動物観察ログへの横展開
- 利用回数制御 / 課金導線

## 13. 非目標

初期段階では次をやりません。

- ChatGPT / Claude のユーザーログイン
- noteへの自動投稿
- 写真そのものの画像解析
- 完全自動保存

## 14. 推奨結論

最初に作るべきものはこれです。

- `フィルム現像後にAIでライフログ下書きを作る`
- `JournalScreen で編集する`
- `noteに貼りやすいMarkdown / 本文としてコピーする`

これなら今のアプリ思想を壊さず、カメラ体験の延長として自然です。

また、あとで動物カメラに広げるときも、AI部分を作り直さずに済みます。
