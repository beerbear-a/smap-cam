# AI Memory Assist Spec

更新日: 2026-03-15

## 1. 概要

この仕様は、ZOOTOCAM に `AIで思い出を整理する` 機能を追加するための確定仕様です。

AIの役割は `勝手に文章を書くこと` ではなく、ユーザーが残した写真・メモ・テーマをもとに、

- ロール全体の要点整理
- ライフログ下書き生成
- note向け本文の整形

を補助することです。

## 2. 機能名

アプリ内の正式名称は `AIで思い出を整理` とする。

設定画面のセクション名は `AI` とする。

ユーザー向け説明文:

`写真やメモをもとに、ロール全体の記録を整理して下書きを作ります。`

## 3. 目的

- フィルム1本を「1日の記録」として文章にしやすくする
- メモの負荷を下げる
- note に貼りやすい文章を作る
- 将来の動物観察カメラにも流用できる基盤にする

## 4. MVPの範囲

MVPでやること:

- フィルム現像後に AI 下書きを生成できる
- Journal 画面で AI 下書きを編集できる
- note向け本文を `コピー / 共有` できる
- 設定画面で AI 利用設定を管理できる
- 自前バックエンド経由で AI を呼ぶ

MVPでやらないこと:

- ChatGPT / Claude のユーザーログイン
- note への自動投稿
- 画像解析を必須にした生成
- 自動生成の強制
- インスタント撮影中のリアルタイム生成

## 5. 体験の原則

- AIは主役ではない
- ユーザーの記録を整理する補助に徹する
- 保存前に必ずユーザーが編集できる
- 断定や創作を避ける
- noteにそのまま貼って読める自然な文章を目指す

## 6. 対象セッション

MVPの対象は `フィルムの現像済みロール` のみ。

対象外:

- 撮影中フィルム
- 現像待ちフィルム
- インスタントセッション

理由:

- 1本のまとまりとして情報が揃っている
- このアプリの思想に最も合う
- note向けの文章単位として自然

## 7. ユーザーフロー

### 7.1 主導線

1. ユーザーがフィルムを現像する
2. 現像完了画面で `AIで思い出を整理` を提案する
3. ユーザーが生成を実行する
4. バックエンドがAI下書きを返す
5. Journal画面で編集する
6. `note向けにコピー` または `共有` する

### 7.2 副導線

Journal画面からいつでも次を実行できる。

- 下書きを生成
- 書き直す
- note向けに整える
- 短くする

## 8. 画面仕様

## 8.1 設定画面

新セクション: `AI`

項目:

- `AIで思い出を整理`
  - 型: トグル
  - 初期値: OFF
- `現像後に提案する`
  - 型: トグル
  - 初期値: ON
  - AI機能OFF時は disabled
- `文章の雰囲気`
  - 値: `日記寄り` / `note寄り`
  - 初期値: `note寄り`
- `画像を送って補助する`
  - 型: トグル
  - 初期値: OFF
  - MVPではUIだけ持たず、非表示でもよい
- `送る情報`
  - 説明文のみ
  - 表示: `場所名、テーマ、メモ、撮影時刻、写真ごとのメモを送ります`

補足文言:

`AIは下書きを作るだけです。保存前に必ず編集できます。`

## 8.2 現像完了画面

現像完了後のアクション群に追加:

- `AIで思い出を整理`

押下時:

- AI機能OFFなら設定導線または簡易説明モーダルを表示
- AI機能ONなら生成開始

ローディング文言:

`ロール全体の記録を整理しています...`

## 8.3 Journal画面

既存のセッションメモ領域の上または下に `AI下書き` セクションを追加する。

表示要素:

- 見出し: `AI下書き`
- サブ説明: `写真とメモから、あとで見返しやすい文章に整理します`
- 本文エディタ
- タイトル入力欄
- アクション

アクション:

- `生成`
- `書き直す`
- `note向けに整える`
- `短くする`
- `コピー`
- `共有`

AI結果未生成時:

- 空状態テキストを表示

空状態文言:

`このロール全体から、ライフログの下書きを作れます。`

## 9. AI出力仕様

AIの返却形式は構造化する。

必須項目:

- `title`
- `subtitle`
- `intro`
- `body_markdown`
- `body_plain_text`
- `hashtags`
- `social_summary`

本文長さ:

- `body_plain_text`: 350〜900文字
- `social_summary`: 80〜140文字

トーン:

- `日記寄り`
  - 一人称寄り
  - 体験の空気感を重視
- `note寄り`
  - 読みやすさ重視
  - 見出しと流れを整理

## 10. note向け書き出し仕様

MVPでは `noteへ直接投稿` はしない。

提供方法:

- `Markdownをコピー`
- `本文をコピー`
- `共有シートを開く`

`Markdownをコピー` の本文テンプレート:

```md
# {title}

{subtitle}

{intro}

## その日のこと

{body_markdown}

{hashtags_line}
```

`hashtags_line` は `#上野動物園 #フィルム写真` のように1行で返す。

## 11. バックエンド仕様

アプリは直接AIプロバイダを呼ばず、自前APIに送る。

## 11.1 エンドポイント

### `POST /v1/memory-assists/drafts`

用途:

- 新規下書き生成

リクエスト:

```json
{
  "session_id": "session_xxx",
  "tone": "note",
  "include_images": false,
  "session": {
    "title": "上野動物園",
    "location_name": "上野動物園",
    "date": "2026-03-15T12:34:56+09:00",
    "theme": "春のやわらかい光",
    "memo": "午後の光が気持ちよかった"
  },
  "photos": [
    {
      "index": 1,
      "timestamp": "2026-03-15T13:01:00+09:00",
      "subject": "レッサーパンダ",
      "memo": "木の上で少し眠そうだった"
    }
  ]
}
```

レスポンス:

```json
{
  "draft_id": "draft_xxx",
  "provider": "openai",
  "model": "gpt-5-mini",
  "tone": "note",
  "title": "上野で過ごした午後、やわらかな光のロール",
  "subtitle": "レッサーパンダを眺めながら残した春の一本",
  "intro": "午後の上野は、少しだけ光がやわらかく見えた。",
  "body_markdown": "本文...",
  "body_plain_text": "本文...",
  "hashtags": ["上野動物園", "フィルム写真", "春の記録"],
  "social_summary": "上野で過ごした午後のロールを、あとから見返せる形に整理した。"
}
```

### `POST /v1/memory-assists/drafts/{draft_id}/refine`

用途:

- 書き直し
- 短縮
- note寄せ

リクエスト:

```json
{
  "action": "refine_for_note"
}
```

許可する action:

- `rewrite`
- `shorten`
- `refine_for_note`

## 11.2 プロバイダ方針

MVPではバックエンドのデフォルトプロバイダを `OpenAI` にする。

理由:

- 構造化出力と運用のしやすさ
- 今後の画像入力拡張に乗せやすい

ただし実装は抽象化し、将来 `Anthropic` を差し替え可能にする。

## 12. 保存データ仕様

新テーブル案: `ai_lifelog_drafts`

カラム:

- `draft_id TEXT PRIMARY KEY`
- `session_id TEXT NOT NULL`
- `provider TEXT NOT NULL`
- `model TEXT NOT NULL`
- `tone TEXT NOT NULL`
- `title TEXT`
- `subtitle TEXT`
- `intro TEXT`
- `body_markdown TEXT`
- `body_plain_text TEXT`
- `hashtags_json TEXT`
- `social_summary TEXT`
- `source_snapshot_json TEXT`
- `created_at INTEGER NOT NULL`
- `updated_at INTEGER NOT NULL`

制約:

- 1セッションにつき複数ドラフトを許可
- 最新ドラフトをUIで優先表示

## 13. 送信する情報

MVPでAIへ送る情報:

- セッションタイトル
- 場所名
- 日付
- テーマ
- セッションメモ
- 写真枚数
- 写真ごとの subject
- 写真ごとの memo
- 写真ごとの timestamp

MVPで送らないもの:

- 緯度経度
- 元画像
- EXIF
- 位置履歴

## 14. 画像送信方針

MVPでは `画像送信なし` を正式仕様とする。

理由:

- プライバシー説明を簡潔に保てる
- コストと速度を抑えられる
- まずは記録整理用途として十分価値がある

将来拡張:

- `代表3枚だけ送る`
- `インデックスシート1枚を送る`

この順で検討する。

## 15. プロンプト方針

バックエンド側のプロンプト原則:

- 事実として入力された内容だけを使う
- 写真に写っている内容を勝手に断定しない
- subject / memo がない情報を創作しない
- 誇張しすぎない
- noteに貼って違和感のない自然な日本語
- 一人称は薄めにしてもよいが、体験の温度は残す

禁止事項:

- 生物種の断定
- 感情の過剰な創作
- 撮影していない場面の補完

## 16. エラーハンドリング

生成失敗時の文言:

`AIでの整理に失敗しました。時間をおいてもう一度お試しください。`

タイムアウト時:

- その場で再試行導線を出す

バックエンド制限時:

- `現在AI整理が混み合っています`

## 17. 利用制限

MVPでは厳しい課金設計は入れない。

暫定制御:

- 1セッションにつき新規生成 3回まで
- refine は 5回まで

この制御はバックエンド側で実施する。

## 18. ログと監査

保存する運用ログ:

- リクエスト日時
- セッションID
- ユーザーID
- 選択トーン
- 利用モデル
- 成功 / 失敗
- レイテンシ

保存しないもの:

- 元画像
- アクセストークンの平文

## 21. この仕様での決定事項

今回の確定事項は次です。

- AIは `思い出整理アシスタント` として扱う
- 自前API経由にする
- MVPはフィルム現像済みロールのみ対象
- MVPでは画像を送らない
- noteには `直接投稿` ではなく `貼りやすい本文の書き出し` を行う
- 設定画面に `AI` セクションを追加する
- Journal画面をAI下書きの編集場所にする
